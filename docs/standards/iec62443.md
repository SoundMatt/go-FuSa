# IEC 62443 — Industrial Automation and Control Systems Security

go-FuSa provides security-level gap assessment and SBOM generation to support
IEC 62443-4-1 and IEC 62443-4-2 compliance.

## Overview

IEC 62443 is the international standard for cybersecurity in industrial
automation and control systems (IACS). It covers:

- **Part 1** — General concepts and terminology
- **Part 2** — Policies and procedures (asset owner)
- **Part 3** — System requirements
- **Part 4** — Component and development requirements

go-FuSa focuses on **Part 4-1** (Secure Product Development Lifecycle) and
**Part 4-2** (Technical Security Requirements for IACS Components).

## Security Levels

| SL | Description |
|----|-------------|
| SL 1 | Protection against casual or coincidental violation |
| SL 2 | Protection against intentional violation with simple means |
| SL 3 | Protection against sophisticated attack with moderate resources |
| SL 4 | Protection against state-sponsored attack with extended resources |

## IEC 62443-4-1 practices

go-FuSa helps satisfy the following SD (Security Development) practices:

| Practice | Description | go-FuSa feature |
|----------|-------------|-----------------|
| SM-6 | Security training | (manual) |
| SR-1 | Security requirements | `gofusa trace` |
| SD-1 | Secure design | `gofusa check` |
| SD-4 | Secure implementation (SBOM) | `gofusa release` |
| SVV-1 | Security testing | `gofusa verify` |
| SVV-3 | Vulnerability testing | `gofusa vuln` |
| DM-1 | Vulnerability management | `gofusa vuln` |

## Rule mapping

| Rule ID | Clause | Description |
|---------|--------|-------------|
| `LINT001` | SD-4 §8.3.1 | No panic (robust error handling) |
| `LINT003` | SD-4 §8.3.2 | All errors handled |
| `LINT004` | SD-3 §8.2.3 | No mutable global state |
| `ANA003` | SD-4 §8.3.4 | Integer overflow detection |

## SBOM (SD-4)

IEC 62443-4-1 SD-4 requires a software bill of materials. Generate with:

```sh
gofusa release --sbom sbom.json
```

The SBOM is in SPDX 2.3 format and lists all Go module dependencies with
versions, allowing downstream consumers to track vulnerability exposure.

## Vulnerability management (DM-1)

```sh
# Scan for known CVEs in dependencies
gofusa vuln

# JSON report for vulnerability management process
gofusa vuln --format json --output vuln-report.json
```

Internally calls `govulncheck` when available; falls back to `go list -json`
dependency enumeration.

## Quick start

```sh
# Security-focused rule check
gofusa check --standard IEC62443

# IEC 62443-4-2 Component Requirement gap report (SL-1 through SL-4)
gofusa iec62443 --sl SL-2

# JSON output for CI gate (exits 1 when gaps exist)
gofusa iec62443 --sl SL-2 --format json --output iec62443-gap-report.json

# Full SBOM + provenance + vuln scan
gofusa release
gofusa vuln
```

## Relationship to ISO 21434

IEC 62443 and ISO 21434 overlap in cybersecurity engineering. For road vehicle
projects that also touch IACS (e.g., V2X infrastructure), consider running
both assessments:

```sh
gofusa iso21434 --cal CAL-3
gofusa unece
gofusa iec62443 --sl SL-2
```
